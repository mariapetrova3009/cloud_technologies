# Лабораторная работа №2  
## Сравнение сервисов Amazon Web Services и Microsoft Azure. Создание единой кросс-провайдерной сервисной модели

### 1. Цель работы  
Научиться классифицировать облачные сервисы Microsoft Azure, построить единую модель, которая подошла бы для обоих провайдеров, чтобы потом можно было сравнивать их услуги по одинаковым категориям

### 2. Исходные данные  
— Файл с данными биллинга Azure  
— Cхема классификации из лабораторной №1 (5 уровней: IT Tower, Service Family, Service Type, Service Sub Type, Service Usage Type).  
— Документация Azure по категориям использования

### 3. Ход работы  
Мы открыли файл с биллингом Azure в Excel и начали сверять каждую строку с документацией. Ориентировались на поля вроде Meter Category, Consumed Service и Meter Name, чтобы понять, что именно было использовано — например, база данных, виртуальная машина или хранилище.

При заполнении колонок мы старались использовать ту же логику, что и для AWS:
- Всё, что связано с базами данных, отнесли к Service Family = Databases, как Amazon RDS в AWS
- Виртуальные машины классифицировали как IaaS, так же, как EC2 в AWS
- Хранилище вынесли отдельно, даже если оно использовалось вместе с базой
- Резервации вычислительных мощностей назвали Reserved Compute, чтобы сохранить аналогию с Reserved Instances в AWS.
важно было не ломать структуру, созданную в первой лабораторной. если в AWS обучение модели машинного обучения шло под AI, Model Training, то в Azure нужно было бы поступить так же

### 4. Примеры классификации сервисов Azure

Вот несколько примеров того, как мы распределили сервисы Azure по своей схеме, используя ту же логику, что и для AWS:

1. Azure SQL Database

 это облачная база данных, которую Microsoft полностью управляет сама. Пользователю не нужно устанавливать СУБД, 
 следить за обновлениями или настраивать сервер, а достаточно просто подключиться и работать с данными. Сервис подходит для хранения 
 информации веб-приложений, мобильных сервисов или внутренних систем компании. Российский аналог: СУБД в Yandex Cloud 
 (Managed PostgreSQL/MySQL) или VK Cloud Managed Databases.
 
 Уровень модели: PaaS (Platform as a Service).

2. Virtual Machines

это виртуальные серверы, которые можно запускать в облаке по запросу. Вы сами выбираете операционную систему, устанавливаете программы и управляете всем,
как на обычном компьютере. Это нужно, когда приложению требуется полный контроль над окружением. 
Аналог в российских облаках: Виртуальные машины в Yandex Compute Cloud или VK Cloud Instances.

Уровень модели: IaaS (Infrastructure as a Service).

3. Azure Data Factory  
это сервис для автоматизации перемещения и обработки данных между разными источниками (базами, файлами, API и т.д.).
Он помогает строить конвейеры (pipelines) для ETL-процессов. Например, собрать данные за день, очистить их и загрузить в
аналитическую базу. Российский аналог: Yandex Data Proc (в связке с другими сервисами) или сервисы оркестрации в VK Cloud.

Уровень модели: PaaS.

4. Azure Web Apps (часть Azure App Service)
Это платформа для быстрого размещения веб-сайтов и веб-API без управления серверами. Достаточно загрузить код приложения — и оно сразу становится доступным в интернете. Подходит для хостинга сайтов, микросервисов и мобильных бэкендов.
Российский аналог: Yandex Cloud App Hosting или VK Cloud Web Apps.

Уровень модели: PaaS.


### 5. Вывод  
Во второй лабораторной работе мы применили свою систему классификации к сервисам Microsoft Azure и убедились, что её можно использовать не только для AWS, но и для других облачных провайдеров. Это важно, потому что компании часто используют несколько облаков, и без единой модели сложно понять, на что именно уходят деньги.  

Работа помогла нам лучше понять, как устроены облачные сервисы, и как их можно группировать по смыслу, а не только по названию. Теперь мы видим, что за разными названиями (например, «EC2» и «Virtual Machines») часто стоит одна и та же идея, то есть аренда вычислительных ресурсов.
